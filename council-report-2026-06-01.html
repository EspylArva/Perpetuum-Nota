LLM Council · Briefing

Review of the stickynotes project

Feedback on a just-shipped MVP of a self-hosted, dockerized Evernote-style notes app (Angular + TipTap / NestJS + Prisma + Postgres) — architecture, features, security, and todos — plus a prioritized plan. Small scale, few hand-picked users; v2 vision is floating windows + an Electron per-note-window desktop client.

Council Verdict

✅ Where the council agrees

  • Harden before v2. 4 of 5 advisors independently called floating windows / Electron / collab "polish on a house with no front-door lock."
  • Cheap network fixes are non-negotiable & fast: login rate-limiting, helmet headers, distinct dev/prod JWT secret.
  • Sharing amplifies every content bug — a javascript: link or unvalidated JSON in a shared note is stored-XSS against whoever opens it.
  • The architecture is a real asset — ProseMirror-JSON + decoupled editor + per-note access predicate is unusually clean.

⚔️ Where the council clashes

Does security matter at this scale? First-Principles argued the real adversary is just trusted users and urged validating the product metaphor first. Peer review overruled it: "blast radius is yourself" is wrong once notes are shared/PUBLIC. Do the ~1-day hardening regardless.

Go for 10x now? Expansionist wants Yjs real-time collab as the headline bet. Reviewers flagged this as the biggest blind spot — building collab + a public API on stored-XSS + no-CSRF multiplies the attack surface. Upside is real, but after hardening.

🔦 Blind spots the peer review caught

  • Object-level authz isn't tested (IDOR). No end-to-end access-matrix test that every note/image endpoint enforces canAccess. Highest-value test to write.
  • Operational recovery is a security issue. No backups + no password reset = a locked-out admin or dead volume bricks the only instance.
  • No detection — no audit/auth logging; a breach would be invisible.
  • Fixes ship unverified — thin tests mean each security patch can silently regress; pair each with a test.
  • Correction: the Postgres port is not host-exposed (only web:8080 is published) — a reviewer's "exposed DB" concern doesn't apply.

🎯 The recommendation

Run a one-week trust pass in three small batches, then re-evaluate before any v2. Don't gold-plate (skip server-side ProseMirror schema validation for now). Don't chase Yjs/Electron yet. Full sequenced plan lives in REVIEW.md.

The one thing to do first
Add a URL-scheme allowlist to link insertion (http/https/mailto) and sanitize hrefs on render. ~1–2 hours, and it closes the one bug already exploitable through the headline sharing feature.

Full advisor responses

The Contrarian downside

No CSRF token with SameSite=Lax is the gap everyone under-rates: Lax does NOT protect you — top-level navigations and auto-submitted forms sail through. httpOnly cookie auth with zero CSRF on every mutating route is the real hole.

But rank honestly: the thing that bites first is the javascript: href — that's stored XSS in a multi-user app where notes are shared. Paste a poisoned link, share PUBLIC, you click it, my script runs in your session. Combined with no CSRF, one shared note compromises accounts. Fix sanitization now (one-line allowlist http/https/mailto).

Second: no login rate limiting — admin-only signup means few high-value accounts and no lockout, a brute-force gift; throttler is an afternoon. What you're rationalizing away: the arbitrary-JSON content endpoint — "type==='doc'" is theater; unvalidated ProseMirror JSON is a second XSS/DoS vector rendered back into other users' browsers.

Cosmetic by comparison: orphaned image cleanup (disk leak), last-write-wins (data loss, annoying not dangerous). Shared dev/prod JWT secret means your repo IS your prod key — rotate it. Order: href sanitize → CSRF → throttler → schema validation → secret rotation → helmet.

The First-Principles Thinker reframe

First-principles question: who is the adversary, and what are they after? Few known users, manual access grants, no open signup, self-hosted behind nginx. That means your real threat surface is overwhelmingly: (1) the trusted users themselves, and (2) whatever's exposed on the public internet if this URL is reachable. Most of your "known gaps" are defenses against attackers you don't have.

CSRF, rate limiting, helmet, JWT secret hygiene matter the instant the URL is public, and they're cheap — do those. But link-href sanitization and content-JSON schema validation are XSS concerns, and your users are hand-picked; the blast radius of a malicious note is yourself. That's near the bottom.

The part nobody's saying: you're not building a SaaS. "Few users, manual grants" is a personal tool with a login. Stop spending architecture budget on multi-tenant permission elegance. The actual thing you're building is "individual windows with content" — v2 floating windows + Electron-per-note IS the product. The real question isn't "what do I harden" — it's "have I validated that the windows-as-notes metaphor is good?" Build the floating-window prototype now. Harden the cheap network-facing stuff. Then go prove the product.

The Expansionist upside

The MVP isn't an Evernote clone. ProseMirror JSON in jsonb plus a decoupled, signal-driven editor and a per-note access predicate is the substrate for a multiplayer document platform. The clone is the demo, not the product.

Highest-leverage bet: real-time collab via Yjs. Content is already ProseMirror, so y-prosemirror is nearly drop-in; your jsonb stays the snapshot/checkpoint while Yjs handles the live CRDT. The moment two cursors land in one note, "sticky notes" becomes a Notion/FigJam competitor. Your per-user grant model already IS the sharing layer collab needs. That's the 10x.

Second, the Electron "each note is an OS window" angle is more unique than collab — nobody ships a notes app where notes are first-class desktop windows you arrange like a real corkboard that also sync live. The undervalued asset: node attrs holding image geometry means your document IS a spatial canvas — the foundation for an infinite whiteboard. Seed now: templates (a note is just seed JSON) and a public document API (addressable JSON → headless CMS, embeds, webhooks). Build order: Yjs → spatial canvas → Electron.

The Outsider fresh eyes

The "I am the admin and I locked myself out" trap. No sign-up means the first thing a self-hoster does is create the admin account — how? If that's a config/CLI step it needs to be dead obvious. Worse: email + password but no password reset means if the admin forgets their own password, the whole app is bricked with no clear way back in. Fix the admin bootstrap and a reset path before anything floating-window.

Sharing has no doorbell. I share a note with you — do you know? Where do you even look for "notes shared with me"? If the answer is "you don't, unless I tell you over text," sharing is decorative. A "Shared with me" view plus a notification badge makes the headline feature actually work.

Deletion with no undo, plus orphaned images. Batch delete + no undo = one wrong click wipes a wall of notes. A trash/recycle bin (restore for 30 days) solves both the accident problem and quietly the leftover-image cleanup.

Backups. A self-hoster's #1 fear is losing everything — one README paragraph on where notes/images live beats any feature. Plus mobile read + self-service password change. Floating windows and a desktop app are polish on a house with no front-door lock and no fire exit.

The Executor ship it

Ship a single "trust hardening" PR this week. Batch into one PR (1-2 days): JWT secret (one-liner, distinct env vars, do first, 5 min); helmet + nginx headers (30 min); throttler on /auth/login (30 min); link href allowlist (genuinely blocking — stored XSS lets a sharer pop your session; scheme-allowlist + sanitize on render, 1-2 hrs); CSRF double-submit cookie (real but contained, half a day). That PR makes it trustworthy to expose.

Second PR same week: autosave 409 (the field already exists in the DTO; client just sends baseContentUpdatedAt + handles 409 — you're losing your own edits today); orphaned images delete-on-delete (skip the periodic sweep cron for now).

Defer, and say so out loud: ProseMirror server validation (low payoff for trusted users; render sanitization covers the actual XSS); e2e tests (add the auth/sharing access-matrix specs after hardening); backups (not a PR — a 10-line pg_dump cron, do it the day you put real notes in). All v2 features: hard defer. Order: secret → headers → throttle → XSS → CSRF, then 409, then image cleanup.

Peer review highlights

What the reviewers converged on
  • Strongest response (unanimous): the Contrarian. Only one to trace the actual kill-chain — shared notes turn javascript: href XSS into cross-account takeover when combined with missing CSRF — and to rank by exploitability rather than checklist order.
  • Biggest blind spots (split): the Expansionist & First-Principles. Expansionist chases collab/whiteboard/public-API on an unhardened base (multiplies attack surface). First-Principles' "blast radius is yourself" premise is factually wrong given sharing/PUBLIC notes — and it sounds reasonable, so it's the more dangerous miss.
  • What all five missed: (1) object-level authorization / IDOR isn't end-to-end tested; (2) operational recovery (backups + password reset) is itself a security/durability risk; (3) no audit logging to detect a breach; (4) thin tests mean fixes ship unverified — pair each with a test.
